# Angular 22 Packaging, Versioning & Distribution — ng-packagr, the exports map, peer deps, semver, npm provenance

> reviewed: 2026-06-30 · tier legend: production-proven | published | experimental

This note grounds **Book 19** (Packaging, Versioning & Distribution), the first book of Volume V (Distribution & Adoption) — it opens the volume. It supplies the version-specific v22 facts behind the book's five moves: the **library build** (ng-packagr + the Angular Package Format), the **consumer contract** (the `exports` map + secondary entry points), **dependency declaration** (peer deps + lockstep), **versioning** (semver + the release/deprecation policy), and **distribution with supply-chain attestation** (npm provenance / SLSA). Book 18 owns *why* the packaging shape matters for bundle cost (tree-shaking, `sideEffects`, secondary entry points as code-split boundaries — see [`angular-22-performance.md`](angular-22-performance.md)); this note grounds *how to publish* — the mirror of the Book 16↔17↔18 why/how boundary. The one keystone the research surfaced: **npm/SLSA provenance attests *build* origin (which repo, which commit, which CI job produced the tarball) — an axis orthogonal to Caelum's *US-origin* claim (who authored/owns the code and its transitive tree).** Provenance is neither necessary nor sufficient for the US-origin claim — a separate, orthogonal axis — so the US-origin attestation is a *separate* artifact shipped inside the package (Book 03 §3.3). Facts were gathered by a 5-facet research fan-out this session; the load-bearing version/builder/field claims below were each fetched from angular.dev (v22.0.4 footers), the ng-packagr repo, the angular-cli spec, docs.npmjs.com, and slsa.dev on 2026-06-30.

## State of the art

### The library build — ng-packagr and the Angular Package Format

- In a generated Angular v22 library project, `ng build <lib>` runs the builder **`@angular/build:ng-packagr`** (the exact `"builder"` string in the library project's `angular.json`) [production-proven] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30, footer v22.0.4+sha-0554bf5)
- This builder **always builds the library with the AOT compiler**; the application build system is esbuild-based (`@angular/build`) while the library build system is ng-packagr-based — two distinct pipelines that can emit different JS from the same TypeScript [production-proven] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30)
- Libraries must be published in **partial compilation** mode — `"compilationMode": "partial"` under `angularCompilerOptions` — producing code "not tied to a specific Angular runtime version," which the consuming application fully compiles during its own build [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30, footer v22.0.4)
- **ng-packagr** "Implements the Angular Package Format," "Bundles your library in FESM2022," and "Creates type definitions (`.d.ts`)," writing output to `dist` [production-proven] (source: https://github.com/ng-packagr/ng-packagr, accessed 2026-06-30)
- The packaging config is **`ng-package.json`**; its schema fields are `$schema`, `dest`, `deleteDestPath`, `keepLifecycleScripts`, `allowedNonPeerDependencies`, `assets`, `inlineStyleLanguage`, and `lib` (with nested `lib.entryFile`, `lib.flatModuleFile`, `lib.cssUrl`, `lib.styleIncludePaths`, `lib.sass`) [published] (source: https://raw.githubusercontent.com/ng-packagr/ng-packagr/main/src/ng-package.schema.json, accessed 2026-06-30). The README example sets `$schema` to `"./node_modules/ng-packagr/ng-package.schema.json"` and `lib.entryFile` to the public-API barrel [production-proven] (source: https://github.com/ng-packagr/ng-packagr, accessed 2026-06-30)
- ng-packagr also exposes a **standalone CLI** — `ng-packagr -p ng-package.json` (its `bin` maps `ng-packagr` → `./src/cli/main.js`) — for building outside the Angular CLI [production-proven] (source: https://github.com/ng-packagr/ng-packagr, accessed 2026-06-30)
- The current stable **ng-packagr is 22.0.0, released 2026-06-03 (Angular v22 GA day), license MIT**; the `main` branch is on prerelease `22.1.0-next.2` [production-proven] (sources: https://github.com/ng-packagr/ng-packagr + https://raw.githubusercontent.com/ng-packagr/ng-packagr/main/package.json, accessed 2026-06-30). Its version tracks Angular majors (22.x ↔ Angular v22) — MIT and US-origin-friendly for a runtime-tier build tool, though ng-packagr runs at **build time only** (dev-tier, per Book 03 §3.3)
- The publish flow is `ng build <lib> --configuration production` → `cd dist/<lib>` → `npm publish` — the `dist/` folder ng-packagr writes **is** the publishable package [production-proven] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30)

### The consumer contract — the `exports` map and secondary entry points

- APF emits a `package.json` with **`"type": "module"`** and a conditional-**`"exports"`** map; each entry point uses two condition keys — **`"types"`** (→ the bundled `.d.ts`) and **`"default"`** (→ the FESM2022 `.mjs`). The v22 spec example is verbatim `".": { "types": "./types/core.d.ts", "default": "./fesm2022/core.mjs" }` [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30, footer v22.0.4)
- The `exports` map also carries **`"./package.json"`** and a **`"./schematics/*"`** glob [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30). The legacy **`"module"`** and **`"typings"`** fields are still emitted but are **deprecated fallbacks** for tooling that does not understand `exports` [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- **Flattened modules** live in **`fesm2022/`** — one `.mjs` per entry point plus a `.map` (`fesm2022/core.mjs`, `fesm2022/testing.mjs`); a FESM is all ESM reachable from one entry point flattened into a single ES2022 module [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30). Bundled **`.d.ts`** live in **`types/`**, one per entry point (recommended since APF v8) [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- APF sets **`"sideEffects": false`** to permit aggressive tree-shaking: "giving the tools more freedom to tree-shake code from the package … should [yield] smaller bundle size" [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30) — the packaging root of the bundle-cost story Book 18 owns
- **Secondary entry points** are each a separate top-level key in `exports` (e.g. `"./testing"`), independently importable and independently tree-shakable; the canonical example is `@angular/core` (`.`) vs `@angular/core/testing` (`./testing`), and Angular Material publishes **each component as its own secondary entry point** [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- ng-packagr **discovers** a secondary entry point by an **`ng-package.json` placed in a subdirectory** of the folder holding the primary `package.json`; that file "can be as simple as `{}` … No name is required. No version is required" — ng-packagr derives the entry-point name from the folder [production-proven] (source: https://raw.githubusercontent.com/ng-packagr/ng-packagr/main/docs/secondary-entrypoints.md, accessed 2026-06-30). Consumers then import the primary via `@my/library` and the secondary via `@my/library/testing` [production-proven] (source: https://raw.githubusercontent.com/ng-packagr/ng-packagr/main/docs/secondary-entrypoints.md, accessed 2026-06-30)
- The same `exports` mechanism can expose **non-JS assets** with a `"sass"` condition — `".": { "sass": "./_index.scss" }`, `"./theming": { "sass": "./_theming.scss" }` — the single door for both code and theme sub-paths [production-proven] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30)

### Dependency declaration — peer dependencies and lockstep versioning

- Angular libraries "should list any `@angular/*` dependencies the library depends on as **peer dependencies**" — not `dependencies` — because "when modules ask for Angular, they all get the exact same module. If a library lists `@angular/core` in `dependencies` … it might get a different Angular module instead, which would cause your application to break" (a duplicate-Angular-instance failure) [published] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30)
- angular.dev does **not** prescribe a specific peer-range *syntax* in prose; the real-world convention comes from Angular's own shipped libraries: `@angular/cdk@22.0.2` declares `"@angular/core": "^22.0.0 || ^23.0.0"` (and the same for `common`/`platform-browser`), i.e. a **multi-major caret-OR range spanning the current + next major** [production-proven] (source: https://registry.npmjs.org/@angular/cdk/latest, accessed 2026-06-30). `@angular/material@22.0.2` mirrors that range, and **pins `@angular/cdk` at exactly `"22.0.2"`** — Material and CDK ship as one unit [production-proven] (source: https://registry.npmjs.org/@angular/material/latest, accessed 2026-06-30)
- **Lockstep:** "As of Angular version 7, the major versions of Angular core and the CLI are aligned … the version of `@angular/core` and the CLI need to be the same" [published] (source: https://angular.dev/reference/releases, accessed 2026-06-30). The releases page does **not** name Material/CDK lockstep in words; that CDK/Material-track-core fact is empirical from the registry (matching `22.0.2`, peer range `^22.0.0 || ^23.0.0` on core) [production-proven] (sources: https://registry.npmjs.org/@angular/cdk/latest + https://registry.npmjs.org/@angular/material/latest, accessed 2026-06-30)
- A library ships **update migrations** so `ng update` can "update your library's dependencies and provide migrations for breaking changes" [published] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30). The package.json key is **`"ng-update"`**, holding `"migrations"` (path to a Schematics collection), `"packageGroup"`, `"packageGroupName"`, and `"requirements"` (a package→min-version map; a mismatch errors the user to update manually first). Each migration entry carries a `"version"` and a `"factory"`; `ng update` runs every factory whose `version` falls in the upgrade span [production-proven] (source: https://raw.githubusercontent.com/angular/angular-cli/main/docs/specifications/update.md, accessed 2026-06-30). The distinct **`"schematics"`** field + **`"ng-add"`** config cover install-time (`ng add`) and generation schematics [published] (source: https://angular.dev/tools/cli/schematics-for-libraries, accessed 2026-06-30)

### Versioning — semver and the release/deprecation policy

- Angular follows **semver** `major.minor.patch`: **major** = "significant new features, some but minimal developer assistance … during the update" (this is where breaking changes are confined); **minor** = "new smaller features … fully backward-compatible; no developer assistance"; **patch** = "Low risk, bug fix" [published] (source: https://angular.dev/reference/releases, accessed 2026-06-30, footer v22.0.4)
- **Cadence:** "A major release every 6 months / 1-3 minor releases for each major release / A patch release and pre-release (`next` or `rc`) build almost every week" — the pre-release train is `-next` then `-rc` (the page names no "beta") [published] (source: https://angular.dev/reference/releases, accessed 2026-06-30)
- **Support window:** each major is "typically supported for 18 months" = **6 months Active** (regular updates/patches) + **12 months LTS** (critical fixes + security patches only) [published] (source: https://angular.dev/reference/releases, accessed 2026-06-30)
- **Deprecation policy:** a deprecated API "is still present in at least the next two major releases (period of at least 12 months). After that, deprecated APIs and features are candidates for removal" [published] (source: https://angular.dev/reference/releases, accessed 2026-06-30)
- **Peer-dependency carve-out:** "In minor releases, we update peer dependencies by expanding the supported versions, but we do not require projects to update these dependencies until a future major version" — the mechanism behind the `^22 || ^23` widening [published] (source: https://angular.dev/reference/releases, accessed 2026-06-30)
- **Experimental / Developer Preview** APIs are exempt: the versioning/deprecation policy "do[es] not apply to APIs marked as Developer Preview [or experimental]. Such APIs can change at any time, even in new patch versions" [experimental] (source: https://angular.dev/reference/releases, accessed 2026-06-30)

### Publishing with attestation — npm provenance and SLSA

- **`npm publish --provenance`** lets you "publicly establish where a package was built and who published a package," producing a **provenance attestation** (a public link to the package's source and build instructions) plus a registry-generated **publish attestation** [production-proven] (source: https://docs.npmjs.com/generating-provenance-statements/, accessed 2026-06-30, last edited 2026-05-04)
- The provenance statement attests, from the CI environment: **source repository URI + commit SHA, the build config/workflow file, build-environment details, and the tarball's cryptographic digest**, signed via the CI job's **OpenID Connect (OIDC)** token [published] (source: https://github.blog/security/supply-chain-security/introducing-npm-package-provenance/, accessed 2026-06-30, published 2023-04-19)
- **Requirements:** a supported cloud CI/CD provider on a cloud-hosted runner — **GitHub Actions** or **GitLab CI/CD** are named — with **OIDC**. On GitHub Actions the workflow needs **`permissions: id-token: write`**, a GitHub-hosted runner (`runs-on: ubuntu-latest`), and `npm publish --provenance` [production-proven] (source: https://docs.npmjs.com/generating-provenance-statements/, accessed 2026-06-30). Flag equivalents: env `NPM_CONFIG_PROVENANCE=true`, `package.json` `"publishConfig": { "provenance": true }`, or `.npmrc` `provenance=true` [production-proven] (source: https://docs.npmjs.com/generating-provenance-statements/, accessed 2026-06-30)
- **Trusted Publishing** publishes "directly from your CI/CD workflows using OpenID Connect (OIDC) authentication," eliminating long-lived npm tokens with "short-lived, cryptographically-signed tokens"; from GitHub Actions / GitLab CI/CD it "automatically generates and publishes provenance attestations … you don't need to add the `--provenance` flag." Auto-provenance "only applies when publishing from public repositories with public packages"; supported providers are GitHub Actions, GitLab CI/CD, **CircleCI**; **self-hosted runners are not supported** [production-proven] (source: https://docs.npmjs.com/trusted-publishers/, accessed 2026-06-30, last updated 2026-06-04)
- The attestation is **signed by Sigstore public-good servers and logged in a public transparency ledger**: **Fulcio** issues a short-lived (~10-minute) X.509 signing certificate from the OIDC token, and **Rekor** is the tamper-evident transparency log [production-proven / published] (sources: https://docs.npmjs.com/generating-provenance-statements/ + https://github.com/npm/provenance, accessed 2026-06-30). Published packages show a **provenance badge** on npmjs.com; consumers verify with **`npm audit signatures`** (npm 9.5.0+) [published] (source: https://github.blog/security/supply-chain-security/introducing-npm-package-provenance/, accessed 2026-06-30)
- npm provenance achieves **SLSA Build Level 2** [published] (source: https://slsa.dev/blog/2023/05/bringing-improved-supply-chain-security-to-the-nodejs-ecosystem, accessed 2026-06-30). SLSA provenance is "the verifiable information about software artifacts describing where, when and how something was produced" — an in-toto attestation (predicate type `https://slsa.dev/provenance/v1`) capturing the build platform identity, inputs, and execution [published] (source: https://slsa.dev/spec/v1.0/provenance, accessed 2026-06-30)
- **Build origin ≠ legal/geographic origin.** Every attested field is a *build-origin* fact (source repo, commit, workflow, CI identity, artifact digest). Provenance proves *this tarball was built from this commit of this repo in this CI job*; it does **not** attest the nationality, legal entity, or location of the maintainers. Caelum's "US-origin-clean" is an orthogonal claim about *who authored/owns the code and its transitive dependencies* — provenance is neither necessary nor sufficient for it (a separate axis), and the US-origin attestation must ship as its own artifact (Book 03 §3.3) [analysis — grounded in the attested-field lists above, no source asserts the negative] (basis: https://docs.npmjs.com/generating-provenance-statements/ + https://github.blog/security/supply-chain-security/introducing-npm-package-provenance/, accessed 2026-06-30)

## Caveats & disputed points

- **`esm2022` is not a v22 `exports` condition key.** The current APF `exports` examples use only `types` + `default` (+ `sass`/`package.json`/`schematics`). The `esm2022` key is widely repeated in blogs but is a pre-consolidation artifact; do not assert it for v22 [basis: the v22 APF page uses only `types`/`default`] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30). Likewise `esm`/`node`/`development`/`production` conditions are not in the v22 spec examples.
- **No documented peer-range syntax.** The `^22.0.0 || ^23.0.0` multi-major form is **observed convention** from Angular's shipped CDK/Material (registry-verified), *not* an angular.dev prose recommendation. Present it as convention, not guidance.
- **The `ng-update` field schema is from the angular-cli repo spec** (`docs/specifications/update.md`, `main` branch) — authoritative but **undated and not v22-tagged**; the angular.dev "Creating libraries" / "Schematics for libraries" pages describe update schematics conceptually and show only `schematics`/`ng-add` JSON, not the `ng-update` block. Verify the exact field set against the installed CLI at M0.
- **Provenance private-package behavior is unconfirmed.** Docs are explicit only that *trusted-publishing auto-provenance* is public-repo/public-package only; whether the manual `--provenance` flag works for a private package is not stated. (Moot for Caelum — it is public — but flagged.)
- **Predicate version emitted today** (SLSA v0.2 vs v1.0) is unconfirmed from a fetched page; the npm/provenance repo references both. The **npmjs.com "Provenance" panel wording** was not re-fetched from a live package page this session.
- **Material-per-component entry points** are asserted by the APF page and corroborate the secondary-entry-point pattern (directly verified via `@angular/core/testing`); a real `@angular/material/package.json` was not fetched this session (npmjs.com HTML returned 403 — registry JSON was used for CDK/Material peer data instead).

## Caelum feasibility path

1. **Package shape.** Ship `@caelum/*` (or a single `caelum` package with per-component secondary entry points) built by `@angular/build:ng-packagr` in partial mode → APF output (FESM2022 + `types/` + `exports` with `types`/`default`, `"sideEffects": false`). One secondary entry point per component/adapter mirrors Material and gives consumers per-component tree-shaking (Book 18 §3.2/§3.3; Book 12 §3.4 for the `provide*()` shape).
2. **Peer deps, not deps.** `@angular/core|common|forms|platform-browser|cdk` (+ `@angular/material` where the Direct layer wraps it) as `peerDependencies` with the `^22.0.0 || ^23.0.0` multi-major range; nothing Angular in `dependencies`. The three adapter libs (grid/charts/editor) stay behind their adapters and are **optional** peer deps or lazily loaded (Book 18 §3.4), never hard runtime deps of the core.
3. **Versioning.** Adopt Angular's semver + a **lockstep major** per Angular major (mint Caelum vN when Angular vN lands; features in Caelum minors, fixes in patches); ship an `ng-update` migration collection so consumers auto-migrate breaking `cae-*` API changes; honor the ≥2-major deprecation window for the public API.
4. **Distribution with two attestations.** Publish from **GitHub Actions via Trusted Publishing** (OIDC, `id-token: write`, no long-lived token) → npm provenance auto-generated (public repo + public package) → SLSA Build L2 + Sigstore/Rekor badge. **Separately**, ship Caelum's **US-origin provenance attestation** (the transitive-tree license/origin record from Book 03 §3.3/§3.4) *inside* the package (a machine-readable manifest + human-readable doc) — npm provenance and the US-origin attestation are orthogonal and both belong in the release.
5. **Two size gates at release.** The app-side `angular.json` budgets do not measure ng-packagr output; add the **shipped-library per-secondary-entry-point size gate** (Book 18 §3.2, PROJECT_BACKLOG M0-4) to the publish workflow.

## Candidate experiments

- **EXP (packaging-shape):** scaffold a two-component `caelum` library with one secondary entry point per component; verify the emitted `dist/package.json` `exports` map has per-entry `types`/`default`, `"sideEffects": false`, `fesm2022/*.mjs`, `types/*.d.ts`; confirm a consumer app tree-shakes an unused component to zero. Success = one unused component contributes 0 bytes to the consumer's initial bundle (ties to Book 18's size gate).
- **EXP (provenance):** publish a throwaway `@caelum/probe` from a public GitHub Actions workflow via Trusted Publishing; confirm the npm provenance badge + `npm audit signatures` pass + a Rekor entry exists; confirm the shipped US-origin manifest is present in the tarball and is *not* what provenance attests (demonstrating the orthogonality).
- **EXP (ng-update):** author a trivial `cae-*` rename migration in an `ng-update` collection; run `ng update @caelum/core` across a mock major bump; confirm the factory rewrites consumer templates.

## Watch

- **ng-packagr 22.1.x** (`main` is `22.1.0-next.2`) — track new APF/`exports` conditions or secondary-entry-point ergonomics before pinning.
- **Trusted Publishing scope** — whether npm expands OIDC trusted publishing beyond GitHub/GitLab/CircleCI, and any change to the public-only auto-provenance rule.
- **SLSA level** — whether npm provenance moves beyond Build L2, and which predicate version (v0.2/v1.0) the CLI emits by default.
- **Angular packaging schematics** — whether v22.1+ folds the `ng-update`/`ng-add`/secondary-entry-point wiring into first-class `ng generate library` scaffolding (reducing bespoke config).
- **`exports` deprecations** — the `module`/`typings` fallbacks are already deprecated; watch for their removal and for `types`/`default` gaining conditions.
