# Angular 22 platform for a component library — version specifics

> reviewed: 2026-06-29 (Angular Aria section re-verified + directive-level API surface added 2026-06-30) · tier legend: production-proven | published | experimental

Angular 22 went stable **2026-06-03**, *after* the model's training cutoff, so every concrete version/API claim about it is frontier and must be sourced — not recalled. This note grounds the version-specific assertions in **Book 01** (Angular 22 Architecture) and the **M0 workspace slices** (M0-1 scaffold, M0-2 install + provenance scan), and supplies the provenance evidence behind **D-01/D-02** (Material+CDK+Aria stack) and **D-10** (US-origin rule) for the *top level* of the core stack. Stable architectural patterns (standalone components, DI, CVA, CDK overlay/a11y) are not frontier and are written from established knowledge; only the lines below are web-sourced.

## State of the art

### Release, toolchain & support window

- Angular 22.0.0 reached stable release on **2026-06-03** [published] (source: https://blog.angular.dev/announcing-angular-v22-c52bb83a4664, accessed 2026-06-29)
- v22 supported Node.js: `^22.22.3 || ^24.15.0 || ^26.0.0` (Node 22, 24, or 26 at those minimums) [published] (source: https://angular.dev/reference/versions, accessed 2026-06-29)
- v22 requires TypeScript `>=6.0.0 <6.1.0` (TS 6.0.x; anything older than 6.0 is dropped) [published] (source: https://angular.dev/reference/versions, accessed 2026-06-29)
- v22 requires RxJS `^6.5.3 || ^7.4.0` [published] (source: https://angular.dev/reference/versions, accessed 2026-06-29)
- The default build system for new apps is the `application` builder (esbuild + Vite based); the legacy webpack `browser` builder, `@angular-devkit/build-angular` webpack builders, and `@ngtools/webpack` are deprecated in v22 [published] (source: https://angular.dev/tools/cli/build-system-migration, accessed 2026-06-29; corroborated: https://blog.angular.dev/announcing-angular-v22-c52bb83a4664)
- v22 browser support is the Baseline "widely available" set at a chosen date of 2026-05-07 (Chrome/Edge/Firefox/Safari released within ~30 months of that date), reaching roughly 95% of web users [published] (source: https://angular.dev/reference/versions, accessed 2026-06-29)
- Angular ships a major every ~6 months; each major gets ~18 months of support (6 months active + 12 months LTS, the latter critical fixes + security only) [published] (source: https://angular.dev/reference/releases, accessed 2026-06-29)
- v22 toolchain-level removals reported in the release notes: TypeScript < 6.0 dropped, Hammer.js integration removed, `createNgModuleRef` removed (use `createNgModule`), `ComponentFactoryResolver`/`ComponentFactory` removed, `ChangeDetectorRef.checkNoChanges` removed, `provideRoutes()` removed [published] (source: https://github.com/angular/angular/releases/tag/v22.0.0, accessed 2026-06-29)

### Change detection — zoneless by default

- Zoneless change detection is the default since v21; per the guide, "Zoneless is the default in Angular v21+ so you do not need to do anything to enable it" [published] (source: https://angular.dev/guide/zoneless, accessed 2026-06-29)
- The stable provider to configure zoneless is `provideZonelessChangeDetection()` — labelled "stable since v20.2" (it is no longer `provideExperimentalZonelessChangeDetection()`) [published] (source: https://angular.dev/api/core/provideZonelessChangeDetection, accessed 2026-06-29)
- In v22, `ChangeDetectionStrategy.OnPush` is the default for new components ("Component with undefined `changeDetection` property are now `OnPush` by default"), and the old `Default` strategy is renamed `ChangeDetectionStrategy.Eager` [published] (source: https://github.com/angular/angular/releases/tag/v22.0.0, accessed 2026-06-29)
- The official v22 announcement frames OnPush-as-default for new applications as aligning with zoneless being the default [published] (source: https://blog.angular.dev/announcing-angular-v22-c52bb83a4664, accessed 2026-06-29)
- Without zones, change detection is triggered by: updating signals read in templates; `ChangeDetectorRef.markForCheck` (called automatically by `AsyncPipe`); bound template/host event-listener callbacks; `ComponentRef.setInput`; and attaching a view marked dirty [published] (source: https://angular.dev/guide/zoneless, accessed 2026-06-29)
- Zone-coupled `NgZone` APIs are incompatible with zoneless and must be removed: `onMicrotaskEmpty`, `onUnstable`, `onStable`, and `isStable` (always `true` under zoneless); `NgZone.run` / `runOutsideAngular` remain compatible [published] (source: https://angular.dev/guide/zoneless, accessed 2026-06-29)
- Component-author guidance: OnPush "is not required, but it is a recommended step towards zoneless compatibility"; `Default`/`Eager` components must notify Angular via `markForCheck`, signals, or `AsyncPipe` [published] (source: https://angular.dev/guide/zoneless, accessed 2026-06-29)
- Removing zone.js is a manual migration (drop it from `angular.json` polyfills, then `npm uninstall zone.js`) [published] (source: https://angular.dev/guide/zoneless, accessed 2026-06-29)

### Signals, reactivity & forms

- `signal()`, `computed()`, and `untracked()` carry no experimental/preview badge on their v22 API/guide pages — stable core signal APIs in v22 [published] (source: https://angular.dev/guide/signals, accessed 2026-06-29)
- `effect()` is "stable since v20.0"; `linkedSignal()` is "stable since v20.0" [published] (source: https://angular.dev/api/core/linkedSignal, accessed 2026-06-29)
- Signal component APIs are stable in v22: `input()`/`input.required()`, `output()` ("stable since v19.0"), `model()` ("stable since v19.0"), and the signal queries `viewChild`/`viewChildren`/`contentChild`/`contentChildren` ("stable since v19.0") [published] (source: https://angular.dev/api/core/viewChild, accessed 2026-06-29)
- The async-reactivity Resource APIs are newly stabilized in v22: `resource()`, `httpResource()`, and `rxResource()` are each "stable since v22.0" [published] (source: https://angular.dev/api/core/resource, accessed 2026-06-29; corroborated: https://angular.dev/api/common/http/httpResource)
- The v22 announcement states `resource` and `httpResource` are "ready for production today" [published] (source: https://blog.angular.dev/announcing-angular-v22-c52bb83a4664, accessed 2026-06-29)
- **Signal Forms** graduated to stable in v22: the `form()` function is "stable since v22.0"; the API lives in `@angular/forms/signals`, where `form(model, schemaOrOptions?, options?)` returns a `FieldTree<TModel>`, with a `FormField` binding directive and validators like `required()`/`email()` configured via a schema function [published] (source: https://angular.dev/api/forms/signals/form, accessed 2026-06-29; guide: https://angular.dev/guide/forms/signals/overview)
- The Signal Forms overview advises that for existing reactive-forms apps or where strict production-stability guarantees are needed, classic reactive forms "remain a solid choice" — Signal Forms is the new-app default, not a forced migration [published] (source: https://angular.dev/guide/forms/signals/overview, accessed 2026-06-29)
- `ControlValueAccessor` still exists in v22, is labelled stable, and remains the documented bridge for integrating custom form controls — it is not deprecated [published] (source: https://angular.dev/api/forms/ControlValueAccessor, accessed 2026-06-29)
- The `@Input` decorator is not deprecated in v22 and "remains fully supported"; signal `input()` is recommended for new code but the two coexist [published] (source: https://angular.dev/guide/components/inputs, accessed 2026-06-29)

### Angular Material 22, CDK 22 & Angular Aria — stack + provenance

- `@angular/material`, `@angular/cdk`, and `@angular/aria` are all at version **22.0.2**, each peering on `@angular/core: "^22.0.0 || ^23.0.0"` — Material/CDK/Aria release in lockstep with Angular core [published] (source: https://registry.npmjs.org/@angular/material/latest, accessed 2026-06-29)
- `@angular/cdk` 22.0.2 is **MIT**-licensed [published] (source: https://registry.npmjs.org/@angular/cdk/latest, accessed 2026-06-29)
- `@angular/aria` 22.0.2 is **MIT**-licensed and peers on `@angular/cdk: "22.0.2"` (exact) [published] (source: https://registry.npmjs.org/@angular/aria/latest, accessed 2026-06-29)
- `@angular/material` 22.0.2 is **MIT** and pins `@angular/cdk` to the exact version "22.0.2" (not a range), mechanically enforcing the in-lockstep CDK/Material pairing [published] (source: https://registry.npmjs.org/@angular/material/latest, accessed 2026-06-29)
- Official policy: "Angular Material and the CDK follow the same support and release policy as the Angular framework," built by "the Angular Components team [which] is part of the Angular team at Google" — i.e. US-origin (Google, Mountain View) [published] (source: https://raw.githubusercontent.com/angular/components/main/README.md, accessed 2026-06-29)
- **Angular Aria** (`@angular/aria`) is "a collection of headless, accessible directives that implement common WAI-ARIA patterns" — it manages keyboard interactions, ARIA attributes, focus management, and screen-reader support while the developer supplies HTML structure, CSS, and business logic [published] (source: https://angular.dev/guide/aria/overview, accessed 2026-06-29)
- Angular Aria ships **12** headless patterns in 3 groups: Autocomplete, Listbox, Select, Multiselect, Combobox (search/selection); Menu, Menubar, Toolbar (navigation/CTA); Accordion, Tabs, Tree, Grid (content organization) — the overview states "12 total patterns" and lists exactly these twelve; **correction (2026-06-30): an earlier read of this note's prose miscounted them as "13" while the list itself was always twelve** [published] (source: https://angular.dev/guide/aria/overview, accessed 2026-06-30)
- Aria directives are imported per-pattern from `@angular/aria/<pattern>` secondary entry points (e.g. `@angular/aria/listbox`, `@angular/aria/menu`, `@angular/aria/combobox`, `@angular/aria/toolbar`, `@angular/aria/tabs`, `@angular/aria/accordion`, `@angular/aria/tree`, `@angular/aria/grid`) and applied as `ng`-prefixed attribute selectors — `ngListbox`/`ngOption`; `ngMenu`/`ngMenuItem`/`ngMenuTrigger`/`ngMenuContent` (+ `ngMenuBar`); `ngCombobox`/`ngComboboxPopup`/`ngComboboxWidget`; `ngTabs`/`ngTabList`/`ngTab`/`ngTabPanel`; `ngAccordionGroup`/`ngAccordionTrigger`/`ngAccordionPanel`/`ngAccordionContent`; `ngTree`/`ngTreeItem`; `ngToolbar`/`ngToolbarWidget`/`ngToolbarWidgetGroup`; `ngGrid`/`ngGridRow`/`ngGridCell` [published] (source: https://raw.githubusercontent.com/angular/angular/main/skills/dev-skills/angular-developer/references/angular-aria.md, accessed 2026-06-30; corroborated: https://angular.dev/guide/aria/listbox)
- The Listbox API: `ngListbox` accepts `multi` (default `false`), `orientation` (`'vertical'|'horizontal'`, default `'vertical'`), `selectionMode` (`'follow'|'explicit'`, default `'follow'`), `wrap` (default `true`), `typeaheadDelay` (default `500` ms), and `disabled`; selection is a two-way `[(value)]` exposing a `Signal<V[]>`; `ngOption` takes a required `value` [published] (source: https://angular.dev/guide/aria/listbox, accessed 2026-06-30)
- Aria patterns compose **on top of** CDK overlay: the Menu guide opens a menu through `cdkConnectedOverlay` / `OverlayModule`, with the trigger exposing `expanded()` as a signal and submenus wired via a `[submenu]` binding — i.e. Aria supplies the pattern/keyboard/ARIA logic while CDK supplies the floating-UI substrate [published] (source: https://angular.dev/guide/aria/menu, accessed 2026-06-30)
- Aria form integration is via a `[formField]` binding on the pattern; "Select", "Multiselect", and "Autocomplete" are **usage patterns composed from the combobox + listbox directives** rather than separate directive packages — which is why the pattern count (12 named patterns) exceeds the number of distinct directive packages [published] (source: https://raw.githubusercontent.com/angular/angular/main/skills/dev-skills/angular-developer/references/angular-aria.md, accessed 2026-06-30)
- Angular Aria is stable in v22 ("Signal Forms, Asynchronous Signals, and Angular Aria are now stable") [published] (source: https://angular.dev/events/v22, accessed 2026-06-29)
- Official "when to use which": Material for components that "look complete without custom styling"; Aria for accessible interactive components with custom styling / specific design requirements; native HTML for straightforward cases [published] (source: https://angular.dev/guide/aria/overview, accessed 2026-06-29)
- CDK 22 ships these primitive packages (folders under `src/cdk`): a11y, accordion, bidi, clipboard, coercion, collections, dialog, drag-drop, keycodes, layout, listbox, menu, observers, overlay, platform, portal, scrolling, stepper, table, testing, text-field, tree [published] (source: https://api.github.com/repos/angular/components/contents/src/cdk, accessed 2026-06-29)
- The CDK `a11y` package contains focus-trap, live-announcer, focus-monitor, interactivity-checker, key-manager, and aria-describer utilities [published] (source: https://api.github.com/repos/angular/components/contents/src/cdk/a11y, accessed 2026-06-29)
- Material 22 theming is design-token + CSS-custom-property based: the `mat.theme()` mixin takes a color/typography/density map and emits CSS variables; system tokens use the `--mat-sys-` prefix (e.g. `var(--mat-sys-surface)`) [published] (source: https://github.com/angular/components/blob/main/guides/theming.md, accessed 2026-06-29)
- Material 22 token overrides: `mat.theme-overrides()` redefines system-level CSS variables and each component exposes an overrides mixin (e.g. `mat.card-overrides()`); colors use the CSS `light-dark()` function for automatic light/dark switching [published] (source: https://github.com/angular/components/blob/main/guides/theming.md, accessed 2026-06-29)

## Caveats & confidence (read before citing the lines above)

These are honest gaps the research pass flagged — do **not** harden them into Book 01 as settled:

- **The official blog post was hard to fetch.** `blog.angular.dev/announcing-angular-v22-...` 307-redirects through a Medium global-identity wall; the reachable variant carried a `?gi=` suffix, and `angular.dev/events/v22` was used as an official substitute for the same wording. The blog URL is cited bare (no `?gi=`) so it stays canonical; re-verify liveness with `--live` before any report ships.
- **The v22.0.0 removed-API list is summary-derived.** It came from a WebFetch summary of the GitHub release page (the same summarizer once mis-rendered the release *year* as 2024 — corrected here to 2026-06-03 against the blog/registry). Treat the exact removed-symbol names as needing a re-read of the rendered release notes before relying on them in code; the TS 6.0 floor and Hammer.js removal are independently corroborated.
- **"New v22 scaffolds omit zone.js" is unverified.** Primary docs describe zone.js removal as a *manual* migration; only secondary sources (not fetched) claim `ng new` now omits it. Stated as open, not asserted.
- **MDC architecture for v22 is inferred.** The only fetchable confirmation that Material is MDC-based is the v17 docs; the v22 docs are a client-rendered SPA that returned only a title shell. Continuous since v15, so very likely current, but not confirmed from a v22 page.
- **`signal()`/`untracked()` stability is by absence-of-badge**, not an explicit "stable since" string (unlike `effect`/`linkedSignal`/`model`/the resource family, which showed explicit badges).
- **Top-level provenance ≠ transitive provenance.** The npm registry confirms the *direct* license/origin of material/cdk/aria (all MIT, Google). D-05/D-10 require the *full transitive tree* — that is the M0-2/M0-4 provenance-scan slice's job, not settled by this note.

## Caelum feasibility path

Against Caelum's actual stack and the ARCHITECTURE invariants:

- **Stack provenance (top level) clears D-10.** `@angular/core`, `@angular/material`, `@angular/cdk`, `@angular/aria` are all MIT and Google-maintained (US). This is the evidence behind D-01/D-02; it does **not** discharge the transitive-tree obligation (M0-2 provenance scan must walk `npm ls` + a license scan over the whole tree). Adapter libraries (D-07 TanStack / D-08 visx / D-09 Lexical) still get their own M2 sign-off.
- **Author every Caelum component OnPush + signal-driven.** v22 makes OnPush the default and zoneless the default; the platform itself pushes this. A consuming app may be zoneless, so Caelum components must not depend on zone.js to detect changes — they drive change detection through signals / `markForCheck` / `AsyncPipe`, never `NgZone.onStable` & co. This is strong enough to be an **architecture invariant candidate** ("zoneless-compatible: OnPush + signal-driven CD, no zone-coupled APIs") — a reversible, low-risk convention; recommend recording it rather than parking work on it. Cross-refs the existing **Angular lockstep** invariant in `docs/ARCHITECTURE.md` §2.
- **Aria reshapes "build on CDK" (D-02), in our favor.** D-02 already endorses "CDK + Angular Aria headless primitives"; this note shows Aria now ships 12 ready headless patterns covering Autocomplete/Listbox/Select/Multiselect/Combobox/Menu/Menubar/Toolbar/Accordion/Tabs/Tree/Grid. Several components we scoped to hand-build on raw CDK (e.g. tree, tabs, menu, multiselect) now have a first-party *headless behavior* layer. Recommended ordering for built components: **Material (if a styled component exists) → Angular Aria headless pattern (+ token-bridge styling) → raw CDK primitive → bespoke**. This is a refinement of D-02 worth a short `decision`/`D-NN` once code starts, not a contradiction of it.
- **Form-control strategy holds (Books 02/07).** CVA is not deprecated and remains the bridge for custom controls, so Caelum's controls build on `ControlValueAccessor` for maximum compatibility (classic + signal forms consumers). Signal Forms being stable is upside (controls should *also* be pleasant to use under `@angular/forms/signals`), not a dependency we bet the library on — protects R3.
- **Toolchain targets for M0-1.** Scaffold the workspace on Node 22/24/26 + TypeScript 6.0.x; library builds via ng-packagr (Book 19), the `forge` demo app via the `application` (esbuild/Vite) builder. Pin the CDK/Material/Aria trio to the same Angular major (lockstep is mechanically enforced by Material's exact CDK pin).

## Candidate experiments

Each a one-line pre-registration seed (hypothesis · metric · baseline); file as `EXP-NN` tickets via `run_experiment` when M0 reaches them:

- **EXP — zoneless smoke.** Hypothesis: a `cae-*` component authored OnPush + signals updates correctly with zone.js uninstalled. Metric: template reflects a signal change with no zone.js present and no manual `detectChanges`. Baseline: same component under `Eager` + zone.js.
- **EXP — Aria-vs-CDK parity.** Hypothesis: a `cae-listbox` built on `@angular/aria` matches one built on raw `@angular/cdk/listbox` on a11y while costing less code. Metric: axe violations (0 both) + keyboard-scenario pass-rate + LOC. Baseline: the raw-CDK build.
- **EXP — transitive provenance scan.** Hypothesis: the M0 dependency tree is fully MIT-compatible + US-origin. Metric: count of non-permissive licenses and flagged-origin packages from `npm ls` + a license scan over the whole tree (target 0/0). Baseline: the bare `@angular/*` install.
