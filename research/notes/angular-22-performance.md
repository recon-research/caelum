# Angular 22 Performance & Bundle Budgets — the size gate, tree-shaking, lazy loading, zoneless CD, profiling

> reviewed: 2026-06-30 · tier legend: production-proven | published | experimental

This note grounds **Book 18** (Performance & Bundle Budgets), the third book of Volume IV (Quality) — it completes the volume. It supplies the version-specific v22 facts behind the book's four levers: **bundle-size gates** (`angular.json` budgets), **tree-shaking** (the Angular Package Format — the load-bearing one, because Caelum *is* a library), **lazy loading** (`@defer` + standalone routes), and **profiling** (Angular DevTools + Chrome DevTools). Zoneless/OnPush change-detection facts are already sourced in [`angular-22-platform.md`](angular-22-platform.md) and cross-referenced, not re-fetched. The one structural insight the research surfaced: **`angular.json` budgets gate the *application* build (the Forge demo), not the `ng-packagr` *library* build** — so Caelum's shipped-package size gate is a *separate* CI mechanism (see Caveats + feasibility). Facts were gathered by a 6-facet research fan-out plus direct re-fetches this session; the load-bearing version/license claims below were personally re-fetched.

## State of the art

### Bundle budgets — the size gate (application builds)

- Angular 22 configures **size budgets** as a `budgets` array on the application build target in `angular.json`; the builder "warns or reports an error when a given part of the application reaches or exceeds a boundary size that you set" [production-proven] (source: https://angular.dev/tools/cli/build, accessed 2026-06-30)
- The build page is served as **v22** (footer `v22.0.4+sha-0554bf5`), confirming the budget model is current for Angular 22 [production-proven] (source: https://angular.dev/tools/cli/build, accessed 2026-06-30)
- Six budget **`type`** values: `bundle` (a specific named bundle, incl. lazy — use with `name`), `initial` (JS+CSS to bootstrap), `allScript` (all scripts), `all` (whole app), `anyComponentStyle` (any one component stylesheet), `anyScript` (any one script) [production-proven] (source: https://angular.dev/tools/cli/build, accessed 2026-06-30)
- **Threshold fields**: `baseline` (reference; defaults to `0`), `maximumWarning`/`maximumError`, `minimumWarning`/`minimumError`, and `warning`/`error` (each applying to both min and max); sizes accept bytes/`kb`/`mb`/`%` (percent is relative to baseline) [production-proven] (source: https://angular.dev/tools/cli/build, accessed 2026-06-30)
- **The gate**: crossing a `warning` threshold logs a warning but the build **succeeds**; crossing an `error` threshold makes `ng build` **fail** — this is what makes budgets a hard CI gate, not advisory [production-proven] (source: https://angular.dev/tools/cli/build, accessed 2026-06-30)
- Builder **type-level defaults**: `initial` defaults to warning at 500kb / error at 1mb; `anyComponentStyle` defaults to warning at 2kb / error at 4kb [production-proven] (source: https://angular.dev/tools/cli/build, accessed 2026-06-30)

### Tree-shaking & library packaging — the Angular Package Format (the Caelum-central lever)

- Angular libraries are published in the **Angular Package Format (APF)**, producing **FESM2022** ("Flattened ES Modules" of ES2022 code) with `"type": "module"` and an `"exports"` map — the format Angular's own packages use and that "optimiz[es] for smaller bundle sizes and faster build times" [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- APF **strongly recommends `"sideEffects": false`** in `package.json`: "The recommendation is for all packages to claim the side-effect free status by setting the `sideEffects` property to `false`" — this is what enables consumers' bundlers to **tree-shake** unused code [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- Libraries are built in **partial compilation** mode (`"compilationMode": "partial"`), producing code "not tied to a specific Angular runtime version," so the consuming application fully compiles the library during its own build [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- APF supports **secondary entry points** resolved via the `"exports"` field, "enabling fine-grained code splitting and lazy loading"; Angular Material "publishes each component as a separate entrypoint" [production-proven] (source: https://angular.dev/tools/libraries/angular-package-format, accessed 2026-06-30)
- The library build tool is **`ng-packagr`** (the configured builder for a library project), which "ensures that the library is always built with the AOT compiler" [production-proven] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30)
- **Tree-shakable services**: "Services should declare their own providers, rather than declaring providers in the NgModule or a component. Declaring a provider makes that service *tree-shakable*"; for global services, expose a `provideXYZ()` function (and the lightweight-token pattern for optional services) [production-proven] (source: https://angular.dev/tools/libraries/creating-libraries, accessed 2026-06-30)

### Lazy loading — `@defer` deferrable views & standalone routes

- **`@defer` deferrable views** "reduce the initial bundle size of your application by deferring the loading of code that is not strictly necessary for the initial rendering of a page"; the deferred block's components/directives/pipes are "split into a separate JavaScript file and loaded only when necessary" [production-proven] (source: https://angular.dev/guide/templates/defer, accessed 2026-06-30)
- Page served as **v22.0.4**; **only standalone** components/directives/pipes can be deferred ("NgModule-based dependencies load eagerly") [production-proven] (source: https://angular.dev/guide/templates/defer, accessed 2026-06-30)
- `@defer` **triggers**: `on idle` (default), `on viewport`, `on interaction`, `on hover`, `on immediate`, `on timer(...)`, and `when <condition>`; each trigger has a `prefetch` variant that loads the chunk ahead of display; **sub-blocks** are `@placeholder`, `@loading`, `@error` [production-proven] (source: https://angular.dev/guide/templates/defer, accessed 2026-06-30)

### Zoneless change detection & the perf model (cross-referenced, not re-fetched)

- Zoneless is the default in v21+ and **OnPush is the default for new v22 components**; the stable provider is `provideZonelessChangeDetection()` (stable since v20.2); without zones, change detection is driven by signal reads, `markForCheck`/`AsyncPipe`, and bound events — see [`angular-22-platform.md`](angular-22-platform.md) › "Change detection — zoneless by default" (originally sourced from https://angular.dev/guide/zoneless, 2026-06-29)
- Removing `zone.js` is a manual migration (drop it from the `angular.json` polyfills, then `npm uninstall zone.js`) — which removes the polyfill from the initial bundle — see [`angular-22-platform.md`](angular-22-platform.md) (source line: https://angular.dev/guide/zoneless, accessed 2026-06-29). **The size saving is a measure-it, not a memorized number** (see Caveats)

### Virtual scroll & grid-at-scale

- `@angular/cdk` is at **22.0.2, MIT** and is a *shipped* (runtime-tier) dependency; it supplies the virtual-scroll primitives a data grid builds on [production-proven] (source: https://registry.npmjs.org/@angular/cdk/latest, accessed 2026-06-30; corroborated in `angular-22-platform.md` 2026-06-29)
- `CdkVirtualScrollViewport` renders only the items that fit on-screen while sizing the container to the full list height, giving a **bounded DOM-node count independent of dataset size** — the primary lever for tables at scale; `*cdkVirtualFor` replaces `*ngFor` with the same API plus a `trackBy` for stable row identity and view recycling [production-proven] (source: https://raw.githubusercontent.com/angular/components/main/src/cdk/scrolling/scrolling.md, accessed 2026-06-30)
- **Fixed-size** virtual scroll (the `itemSize` directive → `FixedSizeVirtualScrollStrategy`) is the recommended path for uniform-height rows; the **autosize** strategy for variable/unknown heights is **experimental** (lives in `cdk-experimental/scrolling`, does not support `scrollToIndex()`) — so a shippable grid prefers fixed heights, and variable-height rows are a genuine capability gap [experimental] (source: https://github.com/angular/components/blob/main/src/cdk-experimental/scrolling/auto-size-virtual-scroll.ts, accessed 2026-06-30; scrolling.md above)

### Profiling — Angular DevTools & Chrome DevTools (dev-tier)

- The **Angular DevTools Profiler** (page served v22.0.4) records change detection: hover the circle → Start recording → interact → Stop; the **bar chart** shows one bar per CD cycle ("the taller a bar is, the longer the application spent running change detection in this cycle"); the **flame graph** shows the render tree, "each tile … an element on the screen"; a **"Change detection" checkbox** greys "those that did not [re-render], such as `OnPush` components that did not re-render"; profiles export/import as JSON (Save Profile / Choose file) [production-proven] (source: https://angular.dev/tools/devtools/profiler, accessed 2026-06-30)
- For frame-granularity render/scripting time, Angular integrates with the **Chrome DevTools Performance** panel via `ng.enableProfiling()` / `enableProfiling()` (**dev mode only**); its Angular track is color-coded (developer TS vs compiled template vs entry points), shows which components were skipped (OnPush not marked dirty), and flags multiple synchronization passes (state mutated during CD — an anti-pattern) [production-proven] (source: https://angular.dev/best-practices/profiling-with-chrome-devtools, accessed 2026-06-30)

### Bundle-analysis tooling & its (dev-tier) provenance

- `@angular/build` is at **22.0.4, MIT** (Angular/Google, `github.com/angular/angular-cli`); its esbuild-based `application` builder is the default, pinning **esbuild 0.28.1, Vite 7.3.5, Rollup 4.60.2** [production-proven] (source: https://registry.npmjs.org/@angular/build/latest, accessed 2026-06-30)
- `ng build --stats-json` emits a `stats.json` the Angular docs say to analyze at `https://esbuild.github.io/analyze/` — the **first-party, zero-extra-dependency** bundle-analysis path (Caelum's recommended default) [production-proven] (source: https://angular.dev/cli/build, accessed 2026-06-30, via the research fan-out; builder pinned esbuild 0.28.1 above)
- `source-map-explorer` is at **2.5.3, Apache-2.0** (`github.com/danvk/source-map-explorer`); it "analyze[s]…space usage through source maps," so it is **builder-agnostic** — it needs only emitted `.map` files, making it the right fit for the esbuild builder [production-proven] (source: https://registry.npmjs.org/source-map-explorer/latest, accessed 2026-06-30)
- `webpack-bundle-analyzer` **5.3.0, MIT** consumes *webpack* stats and is therefore relevant only to the deprecated webpack builder, **not** Angular 22's default esbuild pipeline (a fit mismatch to note) [production-proven] (source: https://registry.npmjs.org/webpack-bundle-analyzer/latest, accessed 2026-06-30, via the research fan-out)
- **All of the above analysis tools are dev-tier** — they run in CI/dev and are never shipped in Caelum's published package, so their licenses (incl. Apache-2.0's NOTICE obligation) and any non-US contributors are governed by Book 03 §3.3's *dev-tier* rule, not the runtime US-origin bar (the same reasoning as axe-core's MPL in `a11y-testing-tooling.md`)

## Caveats & confidence (read before citing the lines above)

- **The load-bearing budget/APF/`@defer` facts are firmly first-hand** (directly fetched from angular.dev pages served as v22.0.4), as are the `@angular/build`, `source-map-explorer`, `@angular/cdk`, and profiler records (personally re-fetched). The virtual-scroll `scrolling.md` and the Chrome-DevTools/webpack-analyzer/esbuild-visualizer facts came through the research fan-out from real canonical URLs; the virtual-scroll API is long-stable but `scrolling.md` and the CDK constants were read from the **main branch, not the 22.0.2 release tag** — verify exact default constants (e.g. `itemSize`/`minBufferPx`/`maxBufferPx`) at the tag before asserting numbers.
- **`angular.json` budgets are an *application*-build feature.** They are documented on the application build page and measure the bootstrapped app's bundles; **`ng-packagr` library builds do not apply them.** That Caelum's *shipped* `cae-*` package therefore needs a *separate* size gate (e.g. `size-limit` or `source-map-explorer` thresholds on the dist, tracked per secondary entry point) is an **inference from the docs' framing**, not an explicit doc statement — treat it as a design conclusion to confirm when the library build is wired (M0/M0-4), not a fetched fact.
- **No "zoneless saves N KB" figure is asserted.** The npm `zone.js` tarball size is the whole multi-build package, *not* the per-app gzipped polyfill; any bundle-saving number must come from a measured in-project build (a candidate experiment below), per Book 18's own "profile, don't guess" thesis.
- **`@defer` stability is stated behaviorally, not by a badge.** The v22 page presents `@defer` as the standard initial-bundle-reduction mechanism (production usage), but no explicit "stable since vN" string was fetched — don't assert a stability version.
- **The ng-new generated budget values were not pinned.** The builder *type-level* defaults (initial 500kb/1mb; anyComponentStyle 2kb/4kb) are from the v22 docs and are solid; the exact values a fresh `ng new` writes into `angular.json` (a different mechanism — the schematic) were only read from the CLI main branch, so don't assert the generated numbers without a tag re-fetch.
- **Patch versions diverge across `@angular/*`.** `@angular/cdk` is 22.0.2 while `@angular/core`/`@angular/build` are 22.0.4 — pin each package independently; do not assert one "22.0.4 everywhere."
- **Under-covered dimensions (honest gaps, not in scope of the fetched facts):** SSR/**incremental hydration** (`@defer`'s `hydrate` triggers), **router preloading** strategies, **NgOptimizedImage**/font/critical-CSS, and **runtime** budget gates (Lighthouse-CI / Web-Vitals LCP/INP/CLS) — Book 18 should name these as gaps/watch items, not cover them from memory.

## Caelum feasibility path

- **Two size gates, not one — because Caelum is a library.** The Forge demo app is gated by `angular.json` budgets (initial/anyComponentStyle, error-fails-the-build); the *published* `cae-*` package needs its **own** shipped-size gate in CI (per-secondary-entry-point bytes via `source-map-explorer`/`size-limit` on the dist, or `--stats-json` on a consumer build). This mirrors the library-is-not-an-application principle (Book 01 §2.2) and the runtime-vs-dev provenance split — the app-build gate and the shipped-package gate are different instruments. Resolve the exact library gate at M0-4 (the CI slice).
- **Tree-shaking is a *packaging* discipline Caelum controls at build time.** Ship APF (FESM2022) with `"sideEffects": false`, **one secondary entry point per component/adapter** (so a consumer importing `cae-select` never pulls `cae-data-grid`), partial compilation, and `provide*()`-style tree-shakable services (the exact shape Book 12's adapter DI already uses). This is the packaging half of Book 19; Book 18 owns *why* (bundle cost), Book 19 owns *how to publish*.
- **Author for zoneless + OnPush (already the platform default).** Signal-driven CD means a `cae-*` row/table only re-checks when its own signals change; the profiler's "Change detection" checkbox is the proof (skipped subtrees grey out). This is the perf pay-off of the standing zoneless-compatible invariant candidate (see `angular-22-platform.md`).
- **Grid-at-scale = fixed-size virtual scroll.** Prefer `CdkVirtualScrollViewport` + `itemSize` (fixed row height) for the R1 lever (Book 10 §3.3); variable-height rows are the experimental autosize gap — a bespoke averaging strategy or fixed-height design is the shippable answer.
- **First-party analysis first.** Default to `ng build --stats-json` → esbuild analyzer (no added dependency); reach for `source-map-explorer` (Apache-2.0, dev-tier) for source-map attribution. Avoid `webpack-bundle-analyzer` (webpack-only, wrong for the esbuild builder).

## Candidate experiments

Each a one-line pre-registration seed (hypothesis · metric · baseline); file as `EXP-NN` via `run_experiment` when the M0/M4 perf slice starts:

- **EXP — bundle-size gate in CI.** Wire `ng build --stats-json` + strict budgets into Forge CI; assert an initial-bundle regression past the error threshold exits non-zero. Metric: CI exit code + gzipped initial bytes. Baseline: the first green M0 scaffold build.
- **EXP — per-component tree-shaking delta.** Ship `cae-*` with `sideEffects:false` + one secondary entry point per component; measure a consumer's bundle importing 1 component vs all. Metric: gzipped bytes pulled per component. Baseline: a single barrel-file entry point (expected: worse tree-shaking).
- **EXP — library shipped-size gate.** Prove a per-entry-point size threshold on the `cae-*` dist (source-map-explorer/size-limit) fails CI when a component crosses its budget. Metric: gate exit code + per-entry-point KB. Baseline: no gate.
- **EXP — zoneless bundle + startup delta.** Measure the gzipped main bundle and Lighthouse TBT/LCP with vs without the `zone.js` polyfill. Metric: KB + TBT/LCP. Baseline: the zoned build. (Replaces any memorized "zoneless saves N KB.")
- **EXP — OnPush/signal CD isolation.** OnPush + signal-driven `cae-*` rows vs `Eager` + zone.js; count components entering CD per scripted interaction (profiler "Change detection" checkbox + Chrome track). Metric: components in CD per cycle. Baseline: plain `@Input` + `markForCheck` under zoned CD.
- **EXP — virtual scroll vs full render at scale.** `cdk-virtual-scroll` (fixed `itemSize`) vs plain `@for` at 1k/10k/50k rows. Metric: rendered DOM node count + median frame time (Chrome Performance) + scroll FPS. Baseline: full render.
- **EXP — `@defer` a heavy adapter.** `@defer (on viewport; prefetch on idle)` the D3 chart or Lexical editor adapter. Metric: initial JS chunk KB + count of emitted lazy chunks. Baseline: the same component eager-referenced (also verify the standalone-only constraint by `viewChild`-querying it and confirming it re-enters the initial chunk).
